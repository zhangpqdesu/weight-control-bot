# 架构选择

## 结论

如果目标是“你和女朋友在普通个人微信群里发文字/图片，机器人自动记录”，优先选：

```text
方案 A：Windows 本机微信 + wxauto + SQLite
```

它几乎没有后端，部署成本最低，最符合现在的需求。限制是电脑要开着，微信要登录，且这是 PC 客户端自动化，不是微信官方开放接口。

如果目标是“尽量官方、可部署、稳定”，需要接受入口变化：

```text
方案 B：微信公众号/服务号 + 轻量 Webhook 后端
方案 C：微信小程序 + 云开发/云托管
方案 D：企业微信应用/群机器人
```

普通个人微信群没有一个官方 API 可以让第三方机器人读取群成员发的文字和图片。

## 方案对比

| 方案 | 能接收个人微信群消息 | 需要后端 | 官方合规 | 图片输入 | 适合程度 |
| --- | --- | --- | --- | --- | --- |
| A. PC 微信 + wxauto | 可以，通过本机自动化 | 不需要公网后端 | 非官方开放接口 | 可以 | 最高，适合 MVP |
| B. 公众号/服务号 | 不可以，用户要私聊公众号 | 需要轻量回调服务 | 官方 | 可以 | 稳定，但不是群聊 |
| C. 小程序 + 云开发 | 不可以，用户进小程序记录 | 微信云开发即可 | 官方 | 可以 | 适合做正式 App |
| D. 企业微信群机器人 Webhook | 通常只能推送到群 | 需要你的系统主动推送 | 官方 | 发送侧有限 | 不适合收饮食记录 |
| E. 企业微信应用 | 可做企业微信内消息能力 | 需要回调服务 | 官方 | 可以 | 适合企业微信，不是个人微信 |

## 推荐落地路线

### 第一阶段：无后端 MVP

```text
普通微信群
  -> PC 微信本机监听
  -> 多模态模型估算
  -> 本地 SQLite
  -> 群内回复本餐和今日统计
```

优点：

- 不需要买服务器
- 不需要公网域名
- 不需要微信公众平台审核
- 你们可以继续用原来的微信群

缺点：

- 电脑必须开着
- PC 微信必须在线
- 客户端 UI 变化可能影响自动化

### 第二阶段：可视化和备份

```text
SQLite
  -> 实时或定时同步
  -> 飞书多维表格
```

这样飞书只负责展示和备份，不影响微信记录链路。

如果你愿意接受网络和 token 依赖，也可以直接把飞书多维表格当主数据库。但饮食记录这个场景里，先写 SQLite 再同步更稳：微信侧不会因为飞书 API 临时失败而丢记录。

### 第三阶段：官方轻量后端

如果你希望长期稳定，建议做成：

```text
微信公众号/服务号
  -> 微信消息回调
  -> 轻量后端：FastAPI / 微信云托管 / 腾讯云函数
  -> 多模态模型
  -> SQLite/Postgres/飞书多维表格
```

这条路线的交互会变成“你们各自私聊公众号记录”，不再是个人微信群。它更官方，也更适合长期开着。

## 数据库选择

当前 MVP 使用 SQLite，原因：

- 无后端时最省心
- 查询每日汇总足够快
- 后续迁移到飞书、多维表格、Postgres 都容易

字段已经按结构化记录设计：

- 人：`me` / `gf`
- 时间：`eaten_at`
- 原始输入：文字、图片路径
- 估算结果：热量、蛋白质、碳水、脂肪、膳食纤维、可信度、推理说明

## 模型接口

当前代码默认使用 OpenAI-compatible Chat Completions 格式：

```env
LLM_BASE_URL=https://api.siliconflow.cn/v1
LLM_MODEL=Qwen/Qwen3-VL-8B-Instruct
```

硅基流动提供兼容 OpenAI 协议的 Base URL，当前 MVP 推荐优先使用 `Qwen/Qwen3-VL-8B-Instruct` 控制成本；复杂餐食再切 `Qwen/Qwen3-VL-30B-A3B-Instruct`。

如果不兼容，只需要改：

```text
src/diet_tracker/llm_client.py
```

其他数据库和微信逻辑不用动。
